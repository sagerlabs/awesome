# Agent 进阶路线：不做 OCR 的版本

## 1. 如何评估效果

先做 Eval（评测集），不要凭感觉判断。

最小版本：

- 建 `docs/TEST_CASES.md` 里的 50 个固定问题。
- 每个问题记录期望点：必须命中哪些英雄、装备、阵容、数值边界。
- 每次改 Prompt（提示词）、Alias（黑话映射）、Knowledge（知识库）后跑一遍。
- 统计命中率、幻觉率、无答案率、平均延迟。

面试时可以这样讲：

```text
我不是只看回答顺不顺眼，而是把问题拆成可验证的事实点，用 Eval 观察改动是否真的变好。
```

## 2. 如何降低延迟

不做 OCR 时，主要优化文字链路：

- Fast NLU（快速自然语言理解）：高频问题不走第一轮 LLM。
- Template Answer（模板回答）：高置信度查询直接模板化输出，少走格式化 LLM。
- Cache（缓存）：缓存热门问题、热门阵容、装备查询结果。
- Short Prompt（短提示词）：减少塞给 LLM 的上下文，只给前 3 个候选。
- Streaming（流式输出）：即使总耗时没变，也让首字更快出现。

优先顺序：

```text
先统计慢在哪里 -> 再决定缓存还是少调 LLM -> 最后再换模型
```

## 3. 如何做版本自动更新

当前可行路线：

- `scripts/update_cn_knowledge.py` 负责拉取 MetaTFT 数据并拆分 knowledge。
- 官方公告 URL 作为 Patch Notes（版本公告）输入，生成版本环境知识。
- Makefile 暴露 `make data PATCH_NOTE_URL=...`。
- 后续可加 GitHub Actions（自动化流水线）定时检查版本。

自动更新不是“自动变聪明”，而是：

```text
新版本数据 -> 结构化知识库 -> 固定测试集回归 -> 通过后发布
```

## 4. 如何从查询助手升级成决策教练

不做 OCR 时，先让玩家用少量文字补局面：

```text
阶段、等级、金币、血量、已有英雄、已有装备、当前羁绊、想连胜还是连败
```

然后让 Agent 输出决策，而不是只输出知识：

- 当前能不能冲。
- 什么时候 D 牌。
- 什么时候升人口。
- 装备现在合不合。
- 如果来牌不顺，转哪套。
- 下一回合买什么、卖什么、留什么。

核心变化：

```text
知识查询：这套阵容强不强
决策教练：你现在这把该不该走这套
```

## 5. 不做 OCR 时的产品形态

先做轻量手动输入：

- 快捷按钮：阶段、等级、金币、血量。
- 常用问题：打工、装备、羁绊、几费主C。
- 允许一句话输入：`3-2，6级，40血，有剑魔羊刀，能不能冲海魔人？`

这样虽然不如 OCR 自动，但已经能展示 Agent 的核心能力：结构化理解、知识检索和局面决策。
