# 2026-04-21 工作记录

## 背景

今天围绕云顶之弈 Agent 的知识库更新能力做了一轮推进，核心目标是让项目从“只会基于固定阵容 JSON 回答”，逐步走向“能跟随版本更新、能回答垂直问题、能理解官方版本环境”的结构。

本次工作不是继续实现 ADR002。ADR002 的 contract boundary（公共协议边界）之前已经完成。今天主要新增和完善的是后续知识层能力。

## 本次实际改动

### 1. 更新最新知识库数据

使用 `scripts/update_cn_knowledge.py` 重新生成了当前版本知识库数据。

主要结果：

- 阵容数据更新到新版本。
- 英雄、装备、阵容拆分 JSON 被重新生成。
- 旧版本数据文件被新版本数据替换，因此 `tft/knowledge/data` 下出现较多删除、新增和修改是预期现象。

### 2. 自动生成费用画像

把“几费卡、谁能 C、谁能抗”这类垂直查询需要的 champion profile（英雄画像）纳入自动更新流程。

主要变化：

- `metadata/tft-meta/get_tftmeta_cn.py` 从 MetaTFT 数据中保留 `unit_profiles`。
- `scripts/update_cn_knowledge.py` 自动生成 `tft/knowledge/data/champion_profiles.json`。
- 费用、羁绊、英雄名称不再依赖手写旧版本画像。

这样做的原因是版本更新时所有弈子、费用、羁绊都可能变化，如果画像仍然手写，知识库很快会过期。

### 3. 实现垂直查询和羁绊查询

新增了知识层对“费用、职责、羁绊”的理解。

现在可以支持的问题类型包括：

- 四费卡谁最强？
- 哪些英雄能 C？
- 哪些英雄能当前排？
- 某个羁绊能不能玩？
- 某个羁绊有哪些核心英雄？

相关结构：

- `unit_cost`：费用查询。
- `role_query`：职责查询，例如 carry（主 C）、tank（前排）。
- `matched_champions`：命中的英雄画像。
- `matched_traits`：命中的羁绊画像。

这里的设计重点是：Agent 不直接猜答案，而是先让 knowledge（知识库）返回结构化命中，再由 Agent 组织成玩家能听懂的话。

### 4. 接入官方版本公告

基于腾讯官方版本公告页面新增 patch notes（版本公告）知识层。

本次使用的页面：

- https://lol.qq.com/gicp/news/662/37082726.html

生成结果：

- 新增 `tft/knowledge/data/patch_notes/17.1.json`。
- 解析出版本号、标题、发布时间和 21 个版本段落。
- 包含系统改动、商店概率、战利品、羁绊、英雄、装备等版本环境信息。

相关能力：

- knowledge 层加载官方版本公告。
- contract（公共协议）里新增 `PatchNoteInsight`。
- Agent prompt（提示词）里新增“官方版本环境”上下文。

这样回答阵容问题时，Agent 不只参考榜单数据，也能感知“这个版本到底改了什么”。

## 我应该记住什么

这次不是单纯“多塞一份 JSON”，而是在补齐知识库的更新闭环。

最重要的设计点：

- 知识库更新脚本负责拉取、翻译、拆分、生成画像、解析版本公告。
- knowledge（知识库）负责把数据变成结构化洞察。
- contract（公共协议）负责定义 Agent 和 knowledge 之间能传什么。
- Agent 负责把结构化结果说成人话，而不是自己凭空判断强弱。

如果以后版本又更新，优先思考的是“这个信息能不能进入自动更新流水线”，而不是“我再手写一份数据”。
